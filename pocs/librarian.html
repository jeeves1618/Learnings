Library Management System Requirements

Overview

The web application will be a library management system developed using Java, Spring Boot, Thymeleaf, HTML, and CSS. The system will have two types of users: Librarians and Members. The interface will be responsive, ensuring it is viewable on PCs, tablets, and mobile devices.

1. User Roles & Permissions

1.1 Librarian

Role: The Librarian will have administrative rights within the application.

Permissions:

  • Add new books to the library collection.
  • Update existing book details.
  • View details of all books in the collection.
  • Delete books from the library collection.
  • Add new members to the library.
  • Update existing member details.
  • View details of all members.
  • Delete members from the system.

1.2 Member

Role: Members will have limited access to the system focused on managing their interactions with the library.

Permissions:

  • View the list of available books.
  • Borrow books.
  • Return borrowed books.
  • Pay membership fees online.

2. Functional Requirements

2.1 Librarian Functionalities

2.1.1 Add Book

Form to input book details: Title, Author, ISBN, Category, Quantity, and Availability Status. On submission, the book is added to the library database.

2.1.2 Update Book

List of books with options to edit each book. Form to update the details of the selected book. On submission, the book details are updated in the database.

2.1.3 View Books

List view of all books in the library. Filter and search options for easy navigation. Detailed view of individual books on selection.

2.1.4 Delete Book

Option to delete books from the list view. Confirmation dialog before deletion.

2.1.5 Add Member

Form to input member details: Name, Address, Contact Information, Membership Type, and Fees Paid. On submission, the member is added to the database.

2.1.6 Update Member

List of members with options to edit each member's details. Form to update the selected member's information. On submission, the member's details are updated in the database.

2.1.7 View Members

List view of all members in the library. Filter and search options for easy navigation. Detailed view of individual members on selection.

2.1.8 Delete Member

Option to delete members from the list view. Confirmation dialog before deletion.

2.2 Member Functionalities

2.2.1 Borrow Book

View a list of available books. Select a book to borrow. Confirmation page showing borrowing details. Borrowing transaction is recorded in the database.

2.2.2 Return Book

List of currently borrowed books by the member. Option to return a book. Confirmation page showing return details. Return transaction is recorded in the database.

2.2.3 Pay Membership Fees

Form to pay membership fees online. Integration with a payment gateway for processing. Confirmation page after successful payment. Payment transaction is recorded in the database.

3. Non-Functional Requirements

3.1 Responsiveness

The application must be fully responsive, using Bootstrap to ensure optimal viewing on PCs, tablets, and mobile devices.

3.2 Security

Role-based access control (RBAC) to ensure only authorized users (Librarians and Members) can access specific functionalities. Secure handling of payment transactions. Data validation on both client-side (using JavaScript) and server-side.

3.3 Performance

The application should load quickly and handle up to 1000 simultaneous users without significant performance degradation. Efficient database queries to minimize load times.

3.4 Scalability

The application architecture should allow for easy addition of new features and scaling to support more users.

4. Technical Stack

  • Backend: Java, Spring Boot
  • Frontend: Thymeleaf, HTML, CSS, Bootstrap
  • Database: MySQL/PostgreSQL/H2 (or any other suitable relational database)
  • Payment Gateway: No need for integration with a standard payment gateway like Stripe or PayPal for membership fees. It is enough if we persist in the database.

5. Development & Testing

5.1 Development

The development will follow Agile methodology with iterative sprints. Continuous integration and deployment (CI/CD) pipelines will be set up for automated testing and deployment.

5.2 Testing

Please consider writing test cases as optional for now. Unit tests for backend services using JUnit. Frontend testing with Selenium or equivalent tool. Security testing for role-based access and payment processing. Performance testing to ensure the application meets performance requirements.

6. Deployment

The application will be deployed only in your local machine. The source code should be added to internal github on a daily basis.

7. Documentation

User documentation and technical documentation should be provided for future maintenance and upgrades.
