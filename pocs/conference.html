Conference Room Management System Requirements

Overview

The web application will be a conference room management system developed using Java, Spring Boot, Thymeleaf, HTML, and CSS. The system will have two types of users: Admins and Conference Room Users. The interface will be responsive, ensuring it is viewable on PCs, tablets, and mobile devices.

1. User Roles & Permissions

1.1 Admin

Role: The Admin will have full control over managing conference rooms and their facilities.

Permissions:

  • Add new conference rooms to the system.
  • Update details of existing conference rooms.
  • View details of all conference rooms.
  • Delete conference rooms from the system.
  • Add new facilities to the system.
  • Update existing facilities in conference rooms.
  • View details of available facilities.
  • Delete facilities from conference rooms.

1.2 Conference Room User

Role: Conference room users will have limited access focused on booking and managing their use of the conference rooms.

Permissions:

  • View the list of available conference rooms.
  • Book a conference room for a specific date and time.
  • Cancel an existing booking.
  • Request additional facilities for a booked conference room.

2. Functional Requirements

2.1 Admin Functionalities

2.1.1 Add Conference Room

Form to input conference room details: Name, Location, Capacity, and Available Facilities. On submission, the conference room is added to the system.

2.1.2 Update Conference Room

List of conference rooms with options to edit each room. Form to update the details of the selected conference room. On submission, the room details are updated in the database.

2.1.3 View Conference Rooms

List view of all conference rooms. Filter and search options for easy navigation. Detailed view of individual conference rooms on selection.

2.1.4 Delete Conference Room

Option to delete conference rooms from the list view. Confirmation dialog before deletion.

2.1.5 Add Facility

Form to input facility details: Facility Name, Description, and Associated Conference Rooms. On submission, the facility is added to the system and associated with the relevant rooms.

2.1.6 Update Facility

List of facilities with options to edit each one. Form to update the details of the selected facility. On submission, the facility details are updated in the database.

2.1.7 View Facilities

List view of all facilities available in the conference rooms. Filter and search options for easy navigation. Detailed view of individual facilities on selection.

2.1.8 Delete Facility

Option to delete facilities from the list view. Confirmation dialog before deletion.

2.2 Conference Room User Functionalities

2.2.1 Book Conference Room

View a list of available conference rooms, including their details and facilities. Select a conference room to book for a specific date and time. Confirmation page showing booking details. Booking transaction is recorded in the database.

2.2.2 Cancel Booking

List of the user's current bookings. Option to cancel a booking. Confirmation page showing cancellation details. Cancellation transaction is recorded in the database.

2.2.3 Request Additional Facilities

For a booked conference room, users can request additional facilities. Form to select desired facilities. Confirmation page showing the updated booking with requested facilities. The request is recorded in the database, and an email notification is sent to the admin.

3. Non-Functional Requirements

3.1 Responsiveness

The application must be fully responsive, using Bootstrap to ensure optimal viewing on PCs, tablets, and mobile devices.

3.2 Security

Role-based access control (RBAC) to ensure only authorized users (Admins and Conference Room Users) can access specific functionalities. Secure handling of booking transactions. Data validation on both client-side (using JavaScript) and server-side.

3.3 Performance

The application should load quickly and handle up to 500 simultaneous users without significant performance degradation. Efficient database queries to minimize load times.

3.4 Scalability

The application architecture should allow for easy addition of new features and scaling to support more users and rooms.

4. Technical Stack

  • Backend: Java, Spring Boot
  • Frontend: Thymeleaf, HTML, CSS, Bootstrap
  • Database: MySQL/PostgreSQL/H2 (or any other suitable relational database)
  • Notification System: Integration with an email service to notify admins of booking requests and facility requests. Please use your own internal email address.

5. Development & Testing

5.1 Development

The development will follow Agile methodology with iterative sprints. Continuous integration and deployment (CI/CD) pipelines will be set up for automated testing and deployment.

5.2 Testing

Please consider writing test cases as optional for now. Unit tests for backend services using JUnit. Frontend testing with Selenium or equivalent tool. Security testing for role-based access and transaction handling. Performance testing to ensure the application meets performance requirements.

6. Deployment

The application will be deployed only in your local machine. The source code should be added to internal github on a daily basis.

7. Documentation

User documentation and technical documentation should be provided for future maintenance and upgrades.
